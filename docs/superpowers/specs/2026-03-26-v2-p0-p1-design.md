# UltraMemory v2.0 — P0+P1 Design Spec

> Inspired by [cognee](https://github.com/topoteretes/cognee)'s graph-first architecture.
> Goal: evolve UltraMemory from a vector-retrieval engine into a graph-aware knowledge engine.

## Decision Log

| Question | Decision | Rationale |
|----------|----------|-----------|
| Multi-layer storage strategy | **A: Multi-vector columns** (same table) | Atomic writes, LanceDB native support, simpler consistency |
| Deterministic ID scope | **A: uuid5(scope + text_hash)** | Preserves scope isolation semantics |
| Feedback signal source | **C: Hybrid** (implicit + explicit) | Zero-config by default, explicit tool for power users |
| Entity normalization approach | **A+C: Static table + LLM prompt guidance** | Reliable without embedding-based false merges |
| Relation traversal depth | **A: 1-hop within candidate pool only** | Zero extra IO, sufficient for candidate pool of ~20 |
| Migration strategy | **A+B: Lazy default + eager CLI** | Zero-downtime default, opt-in full backfill |

## P0 Features

### 1. Multi-Layer Vector Fanout

**Files:** `store.ts`, `retriever.ts`, `embedder.ts`, `service.ts`

#### Schema Change

Add 3 vector columns to the `memories` table:

| Column | Source | Purpose |
|--------|--------|---------|
| `vector` | `entry.text` | Primary search (backward compatible) |
| `vector_l0` | `metadata.l0_abstract` | Concept-level matching |
| `vector_l1` | `metadata.l1_overview` | Structure-level matching |
| `vector_l2` | `metadata.l2_content` | Detail-level matching |

#### Write Path (store.ts + service.ts)

```
store() / importEntry():
  1. embedPassage(text)         → vector     (existing)
  2. embedPassage(l0_abstract)  → vector_l0  (new)
  3. embedPassage(l1_overview)  → vector_l1  (new)
  4. embedPassage(l2_content)   → vector_l2  (new)
```

Optimizations:
- If `l0_abstract === text`, reuse `vector` for `vector_l0` (skip redundant embed)
- If `l2_content === text`, reuse `vector` for `vector_l2`
- Use `embedBatchPassage([l0, l1, l2])` to reduce API round-trips
- Smart extraction already produces distinct L0/L1/L2; manual `memory_store` sets L0=L1=L2=text

#### Store API Change (store.ts) — REQUIRED

The existing `vectorSearch()` signature does not accept a column parameter. It must be extended:

```typescript
// Old signature:
async vectorSearch(vector: number[], limit?, minScore?, scopeFilter?, options?): Promise<MemorySearchResult[]>

// New signature — add optional `column` parameter:
async vectorSearch(vector: number[], limit?, minScore?, scopeFilter?, options?, column?: string): Promise<MemorySearchResult[]>
```

When `column` is provided, chain `.column(col)` on the LanceDB query builder:
```typescript
let query = this.table!.vectorSearch(vector).distanceType('cosine');
if (column) query = query.column(column);  // LanceDB VectorQuery.column() method
```

When omitted, behavior is unchanged (searches default `vector` column).

#### Retrieval Path (retriever.ts)

```
hybridRetrieval():
  Existing: vectorSearch(queryVec)                        + bm25Search()
  New:      vectorSearch(queryVec, ..., 'vector_l0')
            vectorSearch(queryVec, ..., 'vector_l1')
            vectorSearch(queryVec, ..., 'vector_l2')

  5 search channels → deduplicate by memory ID → RRF fusion
```

#### 5-Channel RRF Fusion Pseudocode

```typescript
function fuseMultiChannelResults(
  vectorResults: MemorySearchResult[],       // from 'vector' column
  l0Results: MemorySearchResult[],           // from 'vector_l0' column
  l1Results: MemorySearchResult[],           // from 'vector_l1' column
  l2Results: MemorySearchResult[],           // from 'vector_l2' column
  bm25Results: MemorySearchResult[],         // from BM25 FTS
  config: { vectorWeight, l0Weight, l1Weight, l2Weight, bm25Weight, rrfK }
): FusedResult[] {
  // Channel weights (must sum to 1.0):
  // vector=0.35, l0=0.15, l1=0.10, l2=0.10, bm25=0.30

  const channels = [
    { results: vectorResults, weight: 0.35 },
    { results: l0Results,     weight: 0.15 },
    { results: l1Results,     weight: 0.10 },
    { results: l2Results,     weight: 0.10 },
    { results: bm25Results,   weight: 0.30 },
  ];

  // Build per-memory fusion map
  const fusionMap = new Map<string, FusedResult>();  // memoryId → result

  for (const channel of channels) {
    for (let rank = 0; rank < channel.results.length; rank++) {
      const r = channel.results[rank];
      const rrfScore = channel.weight / (config.rrfK + rank + 1);

      const existing = fusionMap.get(r.entry.id);
      if (existing) {
        existing.score += rrfScore;
        // Track best raw scores per channel for trace
      } else {
        fusionMap.set(r.entry.id, { entry: r.entry, score: rrfScore, sources: {} });
      }
    }
  }

  // Existing BM25 high-score floor logic still applies to bm25 channel:
  // if bm25Score >= 0.75: fusedScore = max(fusedScore, bm25Score * 0.92)
  // BM25-only fallback: if no vector results, clamp01(bm25Score, 0.1)

  return [...fusionMap.values()].sort((a, b) => b.score - a.score);
}
```

RRF weight allocation:
```
vectorWeight: 0.35, l0Weight: 0.15, l1Weight: 0.10, l2Weight: 0.10
bm25Weight:   0.30
Total:        1.00
```

Empty vector columns (un-backfilled legacy data) produce empty result arrays — their RRF contribution is simply zero, requiring no special handling.

#### Migration

- **Lazy (default):** On startup, detect old schema → add vector columns. LanceDB `addColumns` with `valueSql` may not support fixed-size-list literals. **Fallback strategy:** If `addColumns` fails for vector columns, use the **table recreation approach**: (1) read all rows, (2) create new table with expanded schema, (3) write rows back with zero-filled vector columns, (4) drop old table, (5) rename new table. Zero-fill vectors use `new Array(dimensions).fill(0)`. This is O(n) but runs once per migration. For tables >10k rows, log a warning suggesting `upgrade --backfill-vectors` instead.
- **Eager (opt-in):** `ultramemory upgrade --backfill-vectors` CLI command iterates all memories, embeds L0/L1/L2 texts, writes back. Cost: N memories × 3 embed calls (batched via `embedBatchPassage`).

### 2. Deterministic Memory IDs

**Files:** `store.ts`, new `utils/deterministic-id.ts`

#### ID Generation

Uses the `uuid` npm package (add to dependencies: `npm i uuid`):

```typescript
import { v5 as uuidv5 } from 'uuid';

// Custom namespace UUID for UltraMemory (generated once, constant)
const ULTRAMEMORY_NAMESPACE = '7f3e8a2b-4c5d-4e6f-8a9b-1c2d3e4f5a6b';

export function deterministicId(scope: string, text: string): string {
  const input = `${scope}\0${text}`;  // NUL separator prevents scope/text ambiguity
  return uuidv5(input, ULTRAMEMORY_NAMESPACE);
}
```

**Why `uuid` package?** Node.js `node:crypto` does not export a `uuid5` function. The `uuid` package is well-maintained (400M+ weekly downloads) and provides a standards-compliant UUID v5 implementation.

#### Write Path Change

```
store():
  Old: id = randomUUID()
  New: id = deterministicId(scope, text)
       → hasId(id) check first (O(1) lookup)
       → If exists: return { action: "duplicate", existingId: id }
```

#### Two-Level Dedup

1. **Deterministic ID exact match** (O(1)) — catches identical scope+text
2. **Vector similarity dedup** (existing 0.98 threshold) — catches near-identical text variants (e.g., trailing punctuation)

The vector dedup remains as a safety net, not the primary mechanism.

#### Compatibility

- Legacy memories retain their random UUIDs (no migration)
- `importEntry()` preserves original ID (no recalculation)
- `getById()` / `delete()` work with any ID format

### 3. Full Scoring Trace

**Files:** `retriever.ts`, `service.ts`

#### ScoringTrace Interface

```typescript
interface ScoringTrace {
  // Stage 1: Raw search
  vectorScore?: number;
  vectorRank?: number;
  bm25Score?: number;
  bm25Rank?: number;
  l0Score?: number;           // NEW: L0 vector search score
  l1Score?: number;           // NEW: L1 vector search score
  l2Score?: number;           // NEW: L2 vector search score

  // Stage 2: Fusion
  rrfFused?: number;

  // Stage 3: Rerank
  rerankScore?: number;
  rerankBlended?: number;     // 60% rerank + 40% original

  // Stage 4-8: Post-processing
  recencyBoost?: number;      // additive delta
  importanceWeight?: number;  // multiplicative factor
  lengthNorm?: number;        // multiplicative factor
  timeDecay?: number;         // multiplicative factor
  lifecycleDecay?: number;    // NEW: DecayEngine composite score
  mmrPenalty?: number;        // NEW: MMR deduction
  relationBoost?: number;     // NEW: P1 relation boost
  feedbackWeight?: number;    // NEW: P1 feedback weight

  // Final
  finalScore: number;
  queryType?: string;
  searchPath?: "hybrid" | "vector-only" | "bm25-only";  // NEW
}
```

#### Activation

- **Default: OFF.** Normal recall does not populate trace (zero overhead).
- **Opt-in:** `recall({ query, debug: true })` populates full trace.
- Propagation: `RecallParams.debug` → `RetrievalContext.debug` → each scoring function checks `ctx.debug` before writing.

#### Implementation Pattern

Each scoring function gains a conditional write:
```typescript
// Example in applyRecencyBoost():
if (ctx.debug) result.scoringTrace.recencyBoost = boost;
```

No changes to scoring logic itself — trace is read-only instrumentation.

#### Output

```typescript
interface RecallResult {
  // ...existing fields...
  scoringTrace?: ScoringTrace;  // only present when debug=true
}
```

### 4. Per-Category Score Thresholds

**Files:** `retriever.ts`, `config.ts`

#### Config Addition

```typescript
// Added to RetrievalConfig
categoryScoreThresholds?: Record<string, number>;
```

#### Default Values

| Category | Threshold | Rationale |
|----------|-----------|-----------|
| profile | 0.25 | Always relevant, low-score hits still valuable |
| preferences | 0.25 | Same as profile |
| entities | 0.30 | Moderate precision needed |
| events | 0.35 | Current default (unchanged) |
| cases | 0.45 | False positives are costly |
| patterns | 0.45 | Same as cases |

#### Application Point

The `hardMinScore` filter is an inline `.filter()` that appears in **3 locations** in `retriever.ts` (once per retrieval method: vectorOnly, bm25Only, hybrid). All 3 must be updated:

```
Old: results.filter(r => r.score >= hardMinScore)
New: results.filter(r => {
       const threshold = categoryThresholds[r.entry.category] ?? hardMinScore;
       return r.score >= threshold;
     })
```

Consider extracting into a named helper `applyCategoryThreshold(results, config)` to avoid triple duplication.

`hardMinScore` remains as the fallback for unknown/unconfigured categories.

#### Backward Compatibility

- Without `categoryScoreThresholds` config: behavior identical to current (global `hardMinScore`)
- Users can override individual categories; unspecified ones fall back to `hardMinScore`

## P1 Features

### 5. Relation-Aware Reranking

**Files:** `retriever.ts`, `smart-metadata.ts`, `smart-extractor.ts`

#### Pipeline Position

Relation boost is applied **before** `hardMinScore` filtering, so borderline memories can be rescued by their relations:

```
Full actual pipeline (retriever.ts):
  reranked → recencyBoost → importanceWeight → lengthNorm
  → applyRelationBoost (NEW) → hardMinScore filter → timeDecay/lifecycleDecay
  → noiseFilter → MMR → return
```

Placing it before `hardMinScore` ensures that a memory at 0.34 (below the 0.35 default) can be boosted to 0.37 by a relation hit and survive the filter.

#### Algorithm

```
For each candidate R in the result pool:
  1. Parse R's metadata.relations[]
  2. For each relation.targetId:
     - If targetId exists in candidate pool (O(1) Set lookup) → count as "hit"
  3. Also check reverse: if any other candidate has relation pointing to R → count as hit
  4. relationBoost = min(hitCount * RELATION_BOOST_PER_HIT, RELATION_BOOST_CAP)
  5. R.score = clamp01(R.score + relationBoost)
```

Constants:
```typescript
RELATION_BOOST_PER_HIT = 0.03   // +3% per relation hit
RELATION_BOOST_CAP     = 0.09   // max +9% (3 hits cap)
```

#### Bidirectional Matching

Relations are stored unidirectionally (A → B), but matching is bidirectional. Before scoring, build a reverse index:

```typescript
const reverseIndex = new Map<string, Set<string>>();  // targetId → Set<sourceId>
for (const r of candidates) {
  for (const rel of r.relations) {
    reverseIndex.get(rel.targetId)?.add(r.id) ?? reverseIndex.set(rel.targetId, new Set([r.id]));
  }
}
```

A candidate's hit count = forward hits (its relations pointing to pool members) + reverse hits (pool members' relations pointing to it).

#### Relation Generation

In `smart-extractor.ts` `processCandidate()`, when dedup decision is `merge`, `support`, or `contextualize`:

```typescript
// Bidirectional relation creation
appendRelation(newMemory.relations, { type: 'related_to', targetId: existingId });
// Also patch existing memory's relations
await store.patchMetadata(existingId, {
  relations: appendRelation(existingMeta.relations, { type: 'related_to', targetId: newMemory.id })
});
```

Relations accumulate naturally through usage. No manual graph construction needed.

#### Performance

- Candidate pool ~20, avg ~2-3 relations each → ~60 Set lookups + ~60 Map operations
- Negligible compared to embedding/reranking latency

### 6. Feedback Weight EMA

**Files:** `smart-metadata.ts`, `decay-engine.ts`, `service.ts`, new `feedback-learner.ts`

#### Metadata Addition

```typescript
// Added to SmartMemoryMetadata
feedback_weight: number;   // 0.0-1.0, default 0.5
```

**`parseSmartMetadata` must also be updated** to include fallback parsing:
```typescript
feedback_weight: clamp01(typeof parsed.feedback_weight === 'number' ? parsed.feedback_weight : 0.5),
```
This ensures legacy memories without the field default to neutral (0.5).

#### EMA Formula

```
new_weight = alpha * signal + (1 - alpha) * old_weight
```

- `alpha = 0.15` (configurable) — smooth weight transitions
- `signal = 1.0` (positive) or `signal = 0.0` (negative)

#### Signal Sources

| Signal | Type | Trigger Point | Signal Value |
|--------|------|---------------|--------------|
| Manual recall hit | Implicit positive | `service.recall(source="manual")` | 1.0 |
| Auto-recall injected & agent referenced | Implicit positive | `after_tool_call` hook detects memory ID in reply | 1.0 |
| Auto-recall injected but not referenced | Implicit negative | Same hook, no reference detected | 0.0 |
| User calls `memory_forget` | Implicit strong negative | `service.forget()` | Set to 0.1 directly |
| Explicit `memory_feedback` tool | Explicit | New MCP tool | helpful→1.0, not→0.0 |

#### Decay Engine Integration

Current: `intrinsic = importance * confidence`

New:
```
feedbackMultiplier = 0.6 + 0.8 * feedback_weight   // range [0.6, 1.4]
intrinsic = importance * confidence * feedbackMultiplier
```

| feedback_weight | feedbackMultiplier | Effect |
|-----------------|-------------------|--------|
| 0.0 (useless) | 0.6 | -40% intrinsic |
| 0.5 (neutral/default) | 1.0 | No change |
| 1.0 (highly useful) | 1.4 | +40% intrinsic |

#### FeedbackLearner Class

```typescript
// feedback-learner.ts
export class FeedbackLearner {
  constructor(
    private store: MemoryStore,
    private alpha: number = 0.15
  ) {}

  async recordPositive(memoryId: string): Promise<void>    // signal=1.0
  async recordNegative(memoryId: string): Promise<void>    // signal=0.0
  async recordExplicit(memoryId: string, helpful: boolean): Promise<void>

  // Internal: getById → parse feedback_weight → EMA → patchMetadata
}
```

No debouncing needed (feedback events are infrequent compared to access events).

#### New MCP Tool

```
memory_feedback(id: string, helpful: boolean)
→ "Recorded positive/negative feedback for memory {id}"
```

Gated behind `enableManagementTools: true`.

### 8. Bidirectional Lifecycle Triggers

**Files:** `service.ts`, `tier-manager.ts`

#### Problem

Currently only auto-recall triggers full lifecycle evaluation (decay scoring + tier promotion/demotion). Manual `memory_recall` tool calls only update `access_count` and `last_accessed_at` — they never trigger tier transitions.

#### Solution

Add lifecycle evaluation to `service.recall()` for all source types:

```typescript
async recall(params: RecallParams): Promise<RecallResult[]> {
  const results = await this.retriever.retrieve({...});

  for (const r of results) {
    const meta = parseSmartMetadata(r.entry.metadata, r.entry);

    // Existing: update access metadata
    await this._store.patchMetadata(r.entry.id, {
      access_count: (meta.access_count ?? 0) + 1,
      last_accessed_at: Date.now(),
    });

    // NEW: lifecycle evaluation (all sources, not just auto)
    if (this.decayEngine && this.tierManager) {
      const decayable = getDecayableFromEntry(r.entry);
      const decayScore = this.decayEngine.score(decayable.memory);
      const transition = this.tierManager.evaluate(decayable.memory, decayScore);
      if (transition) {
        await this._store.patchMetadata(r.entry.id, { tier: transition.toTier });
      }
    }
  }

  return results;
}
```

#### Forget → Negative Feedback on Related Memories

When a memory is deleted via `memory_forget`, propagate negative feedback to its related memories:

```typescript
async forget(params: ForgetParams): Promise<ForgetResult> {
  if (this.feedbackLearner) {
    const entry = await this._store.getById(params.id);
    if (entry) {
      const meta = parseSmartMetadata(entry.metadata, entry);
      for (const rel of meta.relations ?? []) {
        await this.feedbackLearner.recordNegative(rel.targetId).catch(() => {});
      }
    }
  }
  const ok = await this._store.delete(params.id);
  return { ok };
}
```

#### Error Handling Strategy

Tier `patchMetadata` uses **fire-and-forget** (`.catch(() => {})`) matching the existing access_count pattern:

```typescript
if (transition) {
  this._store.patchMetadata(r.entry.id, { tier: transition.toTier }).catch(() => {});
}
```

**Why fire-and-forget?** Tier transitions are idempotent (re-evaluating the same memory produces the same transition). If two concurrent recalls race on the same memory, both will compute the same target tier and the second write is a harmless no-op. Awaiting would add sequential IO latency to every recall result.

#### Performance

- Tier evaluation is pure computation (no IO) — microseconds per memory
- `patchMetadata` only called when tier actually changes (rare event, fire-and-forget)
- Recall results typically 1-5 items, max 20 — negligible overhead
